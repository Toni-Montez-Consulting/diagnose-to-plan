# Practice OS Concept Registry

This registry preserves the named concepts from the source material so they do
not get flattened into generic SaaS language.

| Concept | Meaning | Current DTP support | New source contribution | Next possible artifact |
|---|---|---|---|---|
| AI/software implementation layer | The practice translates business context into working systems using AI, automation, and software where useful. | Practice roadmap, Business Brain, consulting source-of-truth docs. | Makes the company thesis explicit. | Public/internal positioning map. |
| Businesses without a software team | Primary customer shape: owner-led businesses that need implementation discipline without internal engineering capacity. | Client Operating Kits, proof/comms work. | Sharpens audience without narrowing by industry. | Prospect qualification template. |
| Business-to-system translation | Convert messy operations into workflows, specs, tasks, runbooks, and code-ready plans. | DTP draft, work item specs, command contracts, Context Pack template. | Names the core leverage. | Pilot Context Pack on live work. |
| High-resolution input | Better outputs come from complete-enough context, not magic prompts. | Memory optimization, reply intake, client kits, Input Studio template. | Makes input quality a product principle. | Pilot Input Studio on live work. |
| Thought Inbox | Capture raw notes, transcripts, ideas, lessons, and observations before they disappear. | Contextual idea intake, Notion mirror, steward receipts, Thought Inbox template. | Names the first capture surface. | Use on next raw idea/reply. |
| Input Studio | Transform raw input into structured artifacts while preserving voice and intent. | Business Brain command contracts, comms kits, Input Studio template. | Creates a clear module boundary. | Use on next client/source input. |
| Non-blocking clarifying questions | Ask questions that raise resolution; make labeled assumptions when answers are missing. | Planning mode, DTP diagnose, owner action extraction. | Clarifies question behavior and blocking rules. | Clarifying-question template. |
| Assumptions | Explicit labeled assumptions that let work continue without pretending certainty. | Diagnose/plan docs, meeting prep, action extraction. | Makes assumptions a first-class object. | Assumption ledger template. |
| Context packs | Structured package of intent, audience, workflow, systems, constraints, risks, and acceptance criteria. | Work item spec, handoff runbook, proposal drafts, Context Pack template. | Names the package agents need. | Pilot Context Pack on live work. |
| Workflow mapping | Understand triggers, current state, desired state, systems, owner bottlenecks, and first useful build. | Client Operating Kits, CCAAP/Cam/Greg kits. | Makes workflow redesign central. | Workflow Map template. |
| Opportunity scoring | Prioritize work by frequency, pain, revenue proximity, adoption, readiness, complexity, and risk. | Roadmap steward, active queues, Opportunity Score template. | Adds scoring language for client workflows. | Use before the next infrastructure or client-work expansion. |
| Implementation specs | Translate opportunity into project goal, non-goals, users, inputs, outputs, systems, risks, and acceptance criteria. | Work item spec, DTP draft, tm-skills handoff. | Aligns business-to-code translation. | Implementation Spec template refresh. |
| Build tasks | Break specs into executable work with acceptance criteria and priority. | Roadmap backlog and work-item specs. | Links source loop to implementation execution. | Build Task template. |
| Anti-slop review | Review artifacts for specificity, business truth, user intent, assumptions, risk, and handoff quality. | Review checklist, proof gates, eval garden plans. | Gives the review layer a name. | Anti-Slop Reviewer template/evals. |
| Exception/debug register | Track expected vs observed behavior, severity, root cause, fix, and reusable lesson. | Evidence indexes, steward receipts, support patterns, Exception Register template. | Turns failures into structured learning. | Use after the next meaningful miss. |
| Value ledger | Track time saved, revenue impact, quality, risk, adoption, strategic value, and confidence. | Proof packet, metrics placeholders, Value Ledger template. | Makes value tracking core, not an afterthought. | Use on first client sprint closeout. |
| Human-approved memory | Raw capture can be broad, but durable reusable memory needs review. | Memory control plane, memory optimization plan, Memory Review Queue template. | Defines raw/working/decision/pattern/playbook levels. | Use after two Business Brain reset cycles. |
| Reprioritized backlog | Meaningful events should deliberately update priorities. | Roadmap steward and backlog. | Adds explicit reprioritization cadence. | Reprioritization log use. |
| Proportional pushback | User intent stays primary; blocking only for real risk. | AGENTS.md, CLAUDE.md, collaboration rules. | Preserves momentum while respecting safety. | Planning checklist. |
| Operator console | Private working system for Toni, not a flashy SaaS dashboard. | Hosted DTP Phase 0 and Workbench boundary. | Strengthens product posture. | Hosted-DTP UI brief later. |

## Memory Levels

Use these levels from the source material when planning future memory features:

1. Raw capture.
2. Working memory.
3. Decision memory.
4. Pattern memory.
5. Playbook memory.

Durable pattern/playbook memory must be human-approved.
