Status: active operating map for the offer-led practice machine.

Owner: `diagnose-to-plan`

Purpose: compress the consulting workspace into a small number of operating layers without deleting or flattening valuable ideas. This map is the first stop for broad cross-repo, business, proof, offer, Hub, Practice OS, client-lane, or platform-roadmap prompts.

## North Star

The practice machine exists to make Toni's business easier to sell, explain, deliver, improve, and compound.

It is not a monolithic app. It is a staged operating system:

1. Public offer and trust surface.
2. Private Practice OS and Business Brain.
3. Runtime support and intake operations.
4. Reusable skills, prompts, patterns, and templates.
5. Proof and reference lanes from real project work.

Compression does not mean deletion. Every meaningful idea keeps a home, an activation condition, a not-yet reason, and a way back onto the roadmap when the timing is right.

## Machine Loop

The machine works when work leaves behind useful residue:

1. Capture: client replies, project progress, launch lessons, proof candidates, idea sparks, operator observations.
2. Classify: route to offer, DTP, Hub, proof, client cadence, skill, template, repo touch pass, or parked lane.
3. Stage: mark `Now`, `Next`, `Later`, or `Hold`.
4. Deliver: build, document, run, verify, or hand off in the owning repo.
5. Record: leave evidence, decisions, runbooks, proof/redaction state, eval candidates, or steward notes.
6. Promote: move only reviewed, permissioned, redacted, caveated claims toward public proof.
7. Package: translate the strongest proof into offers, case studies, launch packets, patterns, or client-facing operating systems.
8. Learn: update templates, skills, prompts, lanes, and backlog deliberately.

If a cycle does not produce evidence, a decision, a reusable pattern, a proof update, or a clearer "not yet", the machine did not get stronger.

## Activation Rules

Use this map before accepting broad work that touches more than one repo or one operating layer.

| Prompt shape | First routing step | Default action |
|---|---|---|
| "Make the business/site/platform clearer" | Offer-Led Practice Packaging | classify offer, proof, and public-copy implications before editing |
| "Can this become proof?" | Public Proof Promotion Runbook | require evidence source, permission, redaction, reviewer, caveat |
| "Add a portal/dashboard/command room" | Client Command Room fit assessment | prove recurring workflow pain first |
| "Expand Hub" | Hub Runtime Current-State Map | classify surface as live-hosted, local-only, legacy-proxy, planned, or retired |
| "Automate this across repos" | Workspace Operator Runbook + Workspace Command Center V0 | start read-only and repo-local |
| "Remember/use this later" | Practice Memory Control Plane | decide raw, working, decision, pattern, or playbook memory |
| "Google Workspace, Calendar/Meet, Apple Reminders, LLC, business admin, brand kit, mission/vision, offer catalog" | Business Admin OS + Internal Offer Repertoire Catalog | update DTP first, mirror sanitized status to Notion, keep `/admin` as a launcher only |
| "Publish DSE/Microsoft-adjacent proof" | COI/redaction hold | stop until explicit clearance exists |
| "Build hosted DTP" | Hosted DTP Phase 0 | require real records and separate implementation approval |

## Ownership Guardrails

- DTP stays the durable source of truth for practice method, proof, redaction, private kits, Business Brain, and operating maps.
- Consulting stays the public trust surface. It should show proof only after DTP proof promotion gates pass.
- Hub stays runtime support. It may link to DTP, but it must not replace DTP.
- Prompt and registry repos stay narrow. They should not become practice planning docs.
- Project repos keep their own product/client context. DTP may extract lessons, not absorb their source ownership.
- DSE stays boundary-level unless COI, permission, redaction, and reviewer gates unlock deeper proof work.
