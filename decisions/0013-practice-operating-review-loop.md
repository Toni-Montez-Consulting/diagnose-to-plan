# Decision 0013: Practice Operating Review Loop V0

Date: 2026-05-10

Status: Accepted

## Context

DTP now has multiple internal operating surfaces: Kaizen, Practice Evolution,
Memory Steward, Research Steward, Research Arm, Knowledge Base Event Workflows,
Agent Squads, and the Autonomy Readiness Ladder.

Toni explicitly called out a recurring risk: lightweight captures can get made
and then never revisited. He wants ambition preserved while the system keeps
getting better across projects, repos, client work, business operations, and
future agent squads.

## Decision

Add a DTP-owned Practice Operating Review Loop V0.

The loop defines a lightweight daily check and a deeper weekly operating review.
It uses existing DTP commands and steward surfaces to decide what should be
promoted, built, piloted, parked, rejected, or superseded.

## Consequences

- The practice has a deliberate return path for captured ideas.
- Memory and research recommendations can become decisions instead of chat
  noise.
- Autonomy candidates can be reviewed without building runtime behavior first.
- Client lanes stay protected because this loop does not authorize sends,
  calendar changes, proof movement, Notion sync, tool installs, production
  writes, or autonomous workflows.

## Non-Goals

- No autonomous runtime.
- No Notion live sync.
- No public consulting copy changes.
- No client communication.
- No new hosted service.
- No change to Hub, consulting, or sibling repos.

## Follow-Up

Use the review loop after meaningful internal OS work, after major strategy
threads, before returning to client lanes, and after disconnects or long-running
sessions where capture drift is likely.
