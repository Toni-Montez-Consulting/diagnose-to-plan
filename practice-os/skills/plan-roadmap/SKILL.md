# Purpose

Sequence the work so Phase 1 ships something real without absorbing every good idea.

# Inputs Required

- Diagnose memo
- business constraint
- budget or time appetite
- data sensitivity notes

# Steps

1. State Phase 1 in plain language.
2. State what is explicitly not in Phase 1.
3. Name stack/accounts and owner responsibilities.
4. List risks and acceptance criteria.
5. Park useful future ideas under Phase 2.

# Output Format

Use `practice-os/templates/plan-roadmap.md`.

# Failure Modes

Do not flatten a 2-week build into a vague transformation roadmap. Do not promise a future stack as if it is already decided.
